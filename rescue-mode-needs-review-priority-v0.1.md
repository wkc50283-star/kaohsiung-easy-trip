# 備援模式 Needs review 優先清單 v0.1

本文件整理「一路玩高雄 / 高雄不開車自由行助手」備援模式第一版最需要人工確認的交通與風險資料。

本文件不是施工資料，不是 JSON，不接 API，不查即時班次，只作為人工查證優先清單。

## 1. 文件目的

備援模式的核心原則是：

- 不要害旅人越走越遠。
- 如果使用者還想繼續玩，要提供分級選項：
  1. 順路可玩
  2. 可玩但較繞
  3. 不建議但可自行決定

第一版需要先找出最可能影響安全回收、跨區移動與現場風險的資料。未經 TDX 或官方資料確認前，所有路線編號、站牌、方向、班距、末班車、船班、即時到站與是否一定趕得上，一律標 Needs review。

本文件用途：

1. 排定人工查證優先順序。
2. 區分 P0、P1、P2 交通與風險資料。
3. 說明未確認前第一版應如何保守呈現。
4. 避免把未驗證交通資料寫成確定答案。

## 2. 優先排序原則

| 優先級 | 判斷標準 | 處理態度 |
|---|---|---|
| P0 | 會影響旅人能不能安全回收、可能越走越遠、可能錯過回程的資料 | 第一版施工前最需要人工確認；若未確認，只能保守提示，不可給確定交通細節 |
| P1 | 會影響順不順、會不會太繞、是否適合臨時補行程的資料 | 可先用保守旅遊圈規則，但交通細節仍需 Needs review |
| P2 | 影響體驗但不影響安全回收的資料 | 可延後確認，第一版先不輸出細節 |

## 6. 每個項目的查證欄位

未來人工查證每個項目時，至少應補齊下列欄位：

| 欄位 | 說明 |
|---|---|
| 項目名稱 | 要查證的交通或風險項目 |
| 優先級 | P0、P1 或 P2 |
| 影響旅遊圈 | 對應 rescue-zones |
| 影響辨識點 | 對應 rescue-places |
| 影響回收點 | 對應 rescue-hubs |
| 影響備援情境 | 位置、時間、天氣、體力、心情、跨縣市、未知地點 |
| 官方來源名稱 | 例如 TDX、官方運輸網站、輪船網站、台鐵、高鐵、客運業者 |
| 官方來源連結 | 未填前一律 Needs review |
| 查證日期 | 人工確認日期 |
| 查證者 | 負責人工查證的人 |
| 可確認內容 | 已確認的站點、方向、營運資訊或風險描述 |
| 不可確認內容 | 仍不可寫入前台的資料 |
| 第一版呈現方式 | 未接 API 前的保守文案或限制 |
| 是否可轉 JSON | 未經審核不得轉 |
| 備註 | 例外、季節性、天候或活動影響 |

## 7. 不可猜測的資料

以下資料不可猜測，也不可因為網路文章、印象或過去經驗直接寫成確定資訊：

1. 路線編號。
2. 站牌名稱。
3. 上下車方向。
4. 班距。
5. 末班車。
6. 航班／船班。
7. 是否停駛。
8. 是否一定趕得上。
9. 即時到站。
10. 船是否一定有開。
11. YouBike 是否一定借得到或還得到。
12. 跨縣市客運是否一定仍有班次。
13. 台鐵或高鐵是否一定符合回程時間。

未確認前，前台只能輸出保守方向，例如：

- 先回到明確車站或轉運站。
- 以官方公告與現場資訊為準。
- 時間晚、雨天、風大或拖行李時，不建議硬接遠點。
- 需要搭乘客運、渡輪、台鐵或高鐵時，先查官方班次。

## 8. 第一版仍不可做的事

第一版 Needs review 階段仍不可做：

1. 不接 TDX 即時 API。
2. 不查即時班次。
3. 不做即時定位。
4. 不接 Google Maps API。
5. 不建立 JSON。
6. 不寫程式。
7. 不顯示末班車。
8. 不顯示即時到站。
9. 不承諾一定趕得上。
10. 不推薦未驗證店家。
11. 不輸出確定路線編號。
12. 不輸出確定站牌與上下車方向。
13. 不把未知地點自動猜成已知景點。
14. 不把跨縣市點硬接成高雄市區多點玩法。
15. 不修改首頁 planner。
16. 不修改既有網站功能。
