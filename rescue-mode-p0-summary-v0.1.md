# 備援模式 P0 查證摘要 v0.1

本文件整理備援模式第一版 5 個 P0 官方查證結果，作為未來資料建模與功能施工前可引用的安全規則摘要。

本文件不是功能施工，不是 JSON，不接 API，不查即時班次，不改網站。

## 1. 文件目的

備援模式 P0 查證已完成 5 / 5。

本文件目的：

1. 把 5 份 P0 查證結果整理成第一版可施工摘要。
2. 統一旅遊圈、回收方向、風險提醒與官方資訊優先原則。
3. 明確標出第一版可安全使用與不可使用的資料類型。
4. 保留 Needs review，避免把未確認交通資料寫成確定答案。
5. 作為未來 rescue-zones、rescue-hubs 或其他資料建模的前置依據。

最高原則：

- 不要害旅人越走越遠。
- 第一版只做旅遊圈、回收方向、風險提醒、官方資訊優先。
- 未經官方確認，一律標 Needs review。

## 3. 五個 P0 的共同原則

五個 P0 共同規則：

1. 不寫固定路線。
2. 不寫固定站牌。
3. 不寫固定班距。
4. 不寫固定末班。
5. 不寫現在一定有車或船。
6. 不寫一定趕得上。
7. 不接 TDX 即時 API。
8. 不查即時班次。
9. 不做定位功能。
10. 不把跨縣市或遠郊當成高雄市區短線。
11. 不把未知地點硬判成已知景點。
12. 不推薦未驗證店家。
13. 不把備援模式做成交通保證工具。

五個 P0 可共同採用的輸出邏輯：

1. 先判斷使用者在哪個旅遊圈。
2. 判斷是否已跨縣市或遠離市區主幹。
3. 問剩餘時間、天氣、體力、行李、今晚住哪裡。
4. 優先提供安全回收點。
5. 若使用者還想繼續玩，提供三層分級：
   1. 順路可玩
   2. 可玩但較繞
   3. 不建議但可自行決定
6. 若涉及交通細節，標 Needs review 並要求以官方資訊為準。

## 4. P0-1 旗津／鼓山渡輪摘要

第一版可用定位：

- 旗津／西子灣海邊渡輪圈。
- 鼓山輪渡站與旗津輪渡站可作為重要回收方向。
- 西子灣、鹽埕／駁二、美麗島可作為回市區後續方向。

主要風險：

1. 渡輪、船班、班距、停航與末班受天候、公告、現場調度與營運狀態影響。
2. 雨天、風大、熱季中午、拖行李或時間晚時，旗津回收壓力升高。
3. 不可鼓勵旅人雨天硬走海邊或拖行李進旗津。

第一版安全說法：

> 你現在在旗津，建議先把行程收斂在旗津／鼓山渡輪這一圈。若想回市區，先往鼓山渡輪方向回收，再接西子灣、鹽埕或美麗島。渡輪與船班可能受天候、現場調度與公告影響，請以高雄市輪船官方公告與現場資訊為準。

不可寫死：

- 固定船班。
- 固定班距。
- 末班時間。
- 現在一定有船。
- 一定趕得上。

## 5. P0-2 旗山／美濃摘要

第一版可用定位：

- 高雄遠郊文化圈。
- 旗山轉運站可列為旗山主要交通回收點候選。
- 美濃主要交通點或高客美濃站可列為美濃回收點候選。

主要風險：

1. 旗山／美濃不是市區捷運圈。
2. 回高雄主要仰賴客運或轉乘。
3. 時間晚、下雨、拖行李、長輩小孩同行時，不能再補遠點。
4. 客運路線、站牌、方向、班距與末班不可猜。

第一版安全說法:

> 你目前在旗山／美濃遠郊文化圈。若時間晚、天氣不好、拖行李或體力不足，先不要再加景點。請優先回旗山轉運站或美濃主要交通點，確認官方客運班次後，再往高雄市區、左營或住宿方向移動。

不可寫死：

- 搭某路線一定可以回高雄。
- 從美濃一定直達高雄車站。
- 從旗山一定直達高鐵左營。
- 現在一定有車。
- 回高雄後還能順路接旗津、駁二或夜市。

## 6. P0-3 東港／華僑市場摘要

第一版可用定位：

- 屏東跨縣市回收圈。
- 東港轉運站可列為主要交通回收點候選。
- 華僑市場可作為辨識點，但不應被包裝成高雄市區短線。

主要風險：

1. 東港／華僑市場已離開高雄市區旅遊圈。
2. 可能涉及東港轉運站、客運、碼頭、小琉球船班與住宿決策。
3. 小琉球船班不是回高雄的備援路線。
4. 不可把東港硬接成旗津、駁二、瑞豐夜市、蓮池潭或夢時代多點行程。

第一版安全說法：

> 你目前在東港／華僑市場一帶，已經是屏東跨縣市回收圈。若時間晚、天氣不好或拖行李，先不要再接高雄市區景點。請優先回東港轉運站或主要交通點，確認官方客運班次後，再往小港、左營、高雄車站或住宿方向移動。

不可寫死：

- 東港轉運站一定可直回小港、左營或高雄車站。
- 小琉球船班可當成回高雄路線。
- 固定站牌、方向、班距與末班。
- 回高雄後仍可順路補高雄多點。

## 7. P0-4 恆春／墾丁摘要

第一版可用定位：

- 遠距跨縣市圈。
- 恆春轉運站、小灣或主要交通點可作為回收方向候選。
- 第一個問題應是今晚住哪裡、是否一定要回高雄、是否接受只回收不再加景點。

主要風險：

1. 距離高雄遠，回收時間長。
2. 可能牽涉高鐵左營、高雄市區、枋寮、恆春轉運站與墾丁街車。
3. 颱風、強風、豪雨、道路事故與連假人潮會放大風險。
4. 時間晚時，不應回高雄後再推薦市區景點。

第一版安全說法：

> 你目前在恆春／墾丁，已經是遠距跨縣市圈。請先確認今晚住哪裡、是否一定要回高雄，以及是否能接受只回收不再加景點。若時間晚、天氣不好或體力不足，不建議回高雄後再接市區景點。

不可寫死：

- 一定可直達高鐵左營。
- 一定可直達高雄車站。
- 墾丁街車一定能銜接回高雄客運。
- 固定班距、固定末班、固定站牌與方向。
- 回高雄後還能順路接旗津、駁二、瑞豐夜市或夢時代。

## 8. P0-5 屏東市／潮州摘要

第一版可用定位：

- 跨縣市回收圈。
- 屏東車站可列為屏東市主要鐵路回收點候選。
- 潮州車站可列為潮州主要鐵路回收點候選。
- 潮州轉運站可列為客運輔助回收點候選，但不可混用。

主要風險：

1. 雖可透過台鐵或客運回高雄，但仍不是高雄市區短線。
2. 屏東車站、潮州車站、潮州轉運站不能混用。
3. 班次、停靠站、末班、轉乘與是否能銜接高鐵都不可猜。
4. 時間晚、天氣不好或拖行李時，不應再加高雄市區景點。

第一版安全說法：

> 你目前在屏東市／潮州，已經是跨縣市回收圈。若要回高雄，請優先回屏東車站、潮州車站或主要交通點，查台鐵官方時刻後，再往高雄車站、新左營或住宿方向移動。

不可寫死：

- 屏東車站一定有車回高雄。
- 潮州車站一定有車回高雄。
- 一定停高雄車站或新左營。
- 一定能銜接高鐵。
- 客運一定比台鐵適合。

7. 即時到站。
8. 即時營運狀態。
9. 是否現在一定有車或船。

13. 精準交通時間。

15. 未驗證店家推薦。

## 11. 第一版備援輸出共同文案規則

共同文案結構：

1. 先說使用者目前所屬圈型。
2. 說明是否屬於遠郊、海邊、跨縣市或回收圈。
3. 若風險高，先問今晚住哪裡、是否一定回高雄、有沒有行李、體力是否足夠。
4. 先給安全回收方向。
5. 若使用者還想繼續玩，再給分級建議。
6. 最後加官方資訊優先與 Needs review 提醒。

共同短句：

> 先不要急著補景點，先把位置收斂到主要交通點。

> 時間晚、天氣不好、拖行李或體力不足時，優先回收，不建議再跨區加點。

> 交通路線、站牌、方向、班距、末班與營運異動，請以官方資訊與現場公告為準。

> 第一版只提供方向，不提供固定班次或一定趕得上的承諾。

## 13. 是否可以進入下一階段資料建模

王王判斷：可以討論進入下一階段資料建模，但只能先做 Markdown-to-JSON 前的資料規劃，不得直接做功能。

可進入下一階段的理由：

1. 5 個 P0 已完成查證摘要。
2. 共同紅線已明確。
3. 可安全使用的資料類型已明確。
4. 不可使用的資料類型已明確。
5. Needs review 項目已集中。

下一階段若要施工，應先討論：

1. `rescue-zones` 欄位是否足夠。
2. `rescue-hubs` 欄位是否足夠。
3. P0 是否先只轉成靜態 JSON 草案。
4. 是否需要另建人工審核欄位。
5. 是否先建立 schema，不填實際可執行交通資料。

下一階段仍不得直接做：

1. 不接 TDX 即時 API。
2. 不做定位功能。
3. 不做即時班次。
4. 不做前台互動功能。
5. 不修改首頁 planner。

## 14. 王王結論

P0 查證已完成 5 / 5，可以作為備援模式第一版資料建模前的安全依據。

但目前仍只能進入「資料建模討論」或「Markdown-to-JSON 草案規劃」，不能直接進入功能施工。

王王結論：

> P0 可以做，但只能保守做。第一版不要當交通導航，不要裝作知道即時班次，也不要承諾旅人一定趕得上。備援模式的任務不是讓旅人多跑一個點，而是先把人收回安全主幹。要玩可以，但必須分級；要回收，就優先回主要交通點。這才符合「不要害旅人越走越遠」。
