# 備援模式產品規格 v0.1

本文件記錄「一路玩高雄 / 高雄不開車自由行助手」備援模式第一版產品規格。

備援模式核心原則：

- 不要害旅人越走越遠。
- 如果使用者還想繼續玩，要提供分級選項：
  1. 順路可玩
  2. 可玩但較繞
  3. 不建議但可自行決定
- 第一版不接 TDX 即時 API。
- 第一版不做即時定位。
- 第一版不顯示末班車。
- 第一版不承諾一定趕得上。
- 路線編號、站牌、方向、末班車、即時到站不得猜。
- 未經 TDX 或官方資料確認，一律標 Needs review。

## 1. 功能定位

### 備援模式是什麼

備援模式是旅途中出狀況時的安全改道工具。

當使用者自由行現場發生變數，例如走歪、睡過頭、提早玩完、天氣變差、體力下降、心情改變、只想吃美食逛街、跑到外縣市或未知地點時，網站要重新提供安全、舒服、不亂跑的後續方向。

備援模式的第一優先不是補滿景點，而是先把旅人收斂到好轉乘、好休息、好吃飯、好回程的位置。

### 它跟首頁 planner 差在哪裡

| 項目 | 首頁 planner | 備援模式 |
|---|---|---|
| 使用時間 | 出發前 | 旅途中 |
| 核心問題 | 我從哪裡來、玩幾天 | 我現在在哪、還能怎麼辦 |
| 輸出 | 建議行程入口 | 回收點、改道方向、分級建議 |
| 目標 | 開始排高雄玩法 | 不要越走越遠、降低現場風險 |
| 資料重點 | 出發地、天數 | 目前位置、剩餘時間、天氣、體力、是否跨縣市 |

### 它不是什麼

- 不是即時導航。
- 不是 Google Maps 替代品。
- 不是完整 MaaS。
- 不是 AI 自動排行程。
- 不是即時定位服務。
- 不是交通保證工具。
- 不是餐廳推薦排名。
- 不是一定趕得上的承諾工具。

## 2. 使用情境

| 情境 | 使用者狀況 | 備援模式應做什麼 |
|---|---|---|
| 位置備援 | 走錯站、跑到不在原行程的地方 | 先判斷旅遊圈，給最近安全回收點 |
| 時間備援 | 睡過頭、晚到、提早玩完、快回程 | 依剩餘時間縮小行程，不補太遠景點 |
| 天氣備援 | 下雨、太熱、風大、午後雷陣雨 | 優先改室內、商圈、車站周邊 |
| 體力備援 | 長輩累、小孩累、拖行李、走不動 | 優先休息點、少轉乘、少步行 |
| 心情備援 | 不想看景點，只想吃、逛、坐下 | 改成吃喝、百貨、咖啡、商圈短線 |
| 跨縣市備援 | 跑到屏東、東港、恆春、墾丁 | 先問今晚住哪、是否回高雄，不硬接市區景點 |
| 未知地點備援 | 使用者輸入無法辨識的地點 | 不猜，請使用者補附近車站、商圈、地標或行政區 |

## 3. 使用者操作流程

第一版最小流程如下：

1. 使用目前位置

   第一版不做即時定位。可先讓使用者手動選擇或輸入目前附近地點。

2. 定位成功或失敗

   - 成功：對應到 `rescue-places` 或 `rescue-zones`。
   - 失敗：進入未知地點備援，要求補充附近車站、捷運站、商圈、行政區或地標。

3. 選擇目前想怎麼玩

   可選方向：

   - 繼續玩
   - 想吃東西
   - 想逛室內
   - 想休息
   - 想回住宿
   - 想回高雄市區
   - 想去回程點

4. 選擇剩餘時間

   第一版只用粗分：

   - 1 小時內
   - 2 到 3 小時
   - 半天
   - 今天還很完整

   不寫死交通時間。

5. 若跨縣市，追加詢問

   - 今晚住哪裡？
   - 是否一定要回高雄？
   - 是否有大行李？
   - 是否能接受只做回收，不再加景點？

6. 輸出分級建議

   固定輸出三層：

   - 順路可玩
   - 可玩但較繞
   - 不建議但可自行決定

   每則建議至少包含：

   - 建議方向
   - 為什麼
   - 安全回收點
   - 風險提醒
   - Needs review 標記，若涉及未驗證交通資料

## 4. 資料架構

### rescue-zones

用途：定義旅遊圈。

範例旅遊圈包含：

- 美麗島／高雄車站
- 鹽埕／駁二
- 旗津／西子灣
- 左營／巨蛋
- 三多／夢時代
- 旗山／美濃
- 屏東／東港
- 恆春／墾丁

此資料用來判斷使用者目前在哪個圈，以及哪些方向順路、較繞或不建議硬接。

### rescue-places

用途：定義可辨識地點。

範例地點包含：

- 美麗島
- 高雄車站
- 駁二
- 旗津
- 左營
- 東港
- 墾丁

此資料用來把使用者輸入的地點對應到旅遊圈、風險、適合玩法與安全回收點。

### rescue-hubs

用途：定義安全回收點。

範例回收點包含：

- 高雄車站
- 美麗島站
- 左營／高鐵左營
- 三多商圈站
- 屏東車站
- 東港轉運站
- 恆春轉運站

此資料用來輸出「先回到哪裡比較安全」。

若涉及路線編號、站牌、方向、班表、末班車或即時狀態，必須由 TDX 或官方資料確認。

### rescue-rules

用途：定義備援情境規則。

第一版至少包含：

- 位置備援
- 時間備援
- 天氣備援
- 體力備援
- 心情備援
- 跨縣市備援
- 未知地點備援

此資料用來決定輸出優先順序與保守提醒。

### rescue-user-state

用途：記錄使用者當下狀態。

這不是固定資料，而是一次備援判斷中的暫時狀態。

可能包含：

- 目前地點
- 所屬旅遊圈
- 剩餘時間
- 天氣狀態
- 體力狀態
- 心情偏好
- 是否跨縣市
- 是否要回高雄
- 是否有大行李

第一版不做會員、不儲存個人資料、不做定位紀錄。

### rescue-result

用途：定義系統輸出結果。

輸出應包含：

- 順路可玩
- 可玩但較繞
- 不建議但可自行決定
- 安全回收點
- 保守提醒
- Needs review

## 5. TDX 使用邊界

### 第一版不接 TDX

第一版不做：

- 不呼叫 TDX 即時 API。
- 不使用 TDX token。
- 不顯示路線編號。
- 不顯示班次。
- 不顯示末班車。
- 不顯示即時到站。
- 不判斷現在一定趕得上。

第一版只做人工資料表、保守規則與 Needs review。

### 第二版可用 TDX 驗證資料

第二版可考慮用 TDX 驗證：

- 公車／客運路線
- 站牌與站位
- 路線方向
- 捷運／輕軌站點與轉乘
- 台鐵站點與路線
- 高鐵左營回程主幹
- 渡輪／航運資料是否涵蓋鼓山旗津或相關航線
- YouBike 站點是否可作輔助移動資料

第二版仍不一定要做即時查詢，可先做官方資料對照與人工審核。

### 第三版才考慮即時資料

第三版才考慮：

- 即時到站
- 即時班表
- 末班車提示
- 航運異常
- 營運警示
- 是否還能回收

若需要保護 TDX token，可能需要後端或 serverless function。這屬於重大事項，必須董事長明確批准。

### 必須官方確認的資料

以下資料不可猜：

- 路線編號
- 站牌名稱
- 上下車方向
- 末班車
- 班距
- 即時到站
- 是否停駛
- 跨縣市客運路線
- 航運船班
- 台鐵／高鐵班表與異常

### 不可猜的輸出

不得輸出：

- 搭哪一班一定可行
- 幾點末班
- 一定來得及
- 這條客運一定有
- 這個站牌一定在附近
- 船一定有開
- YouBike 一定借得到

## 6. 第一版 MVP 範圍

第一版只做：

- 靜態旅遊圈
- 靜態辨識點
- 安全回收點
- 順路／較繞／不建議
- Needs review
- 保守提醒文案

第一版輸出方向範例：

| 狀態 | 輸出方向 |
|---|---|
| 使用者在旗津、下雨 | 不建議硬走海邊，先回鼓山渡輪或西子灣站，改室內 |
| 使用者在左營、還有 2 到 3 小時 | 順路可玩巨蛋／瑞豐，較繞是鹽埕，旗津不建議 |
| 使用者在東港 | 先問是否回高雄與今晚住哪，不直接推薦高雄港邊玩法 |
| 使用者位置未知 | 要求補充車站、商圈、行政區，不猜測 |

## 8. 風險

### 交通資料風險

路線、站牌、方向、末班車、班距、渡輪、跨縣市客運都不能人工猜。若錯誤，可能讓旅人回不了住宿或錯過回程。

### 天氣資料風險

旗津、西子灣、大港橋、蓮池潭、衛武營都會公園在雨天、風大或熱季中午風險高。若只看一般天氣，不看警報，仍可能誤導。

### 跨縣市風險

屏東、東港、恆春、墾丁不是高雄市區短線。若旅人已跨縣市，備援模式應優先回收，不應鼓勵再接高雄多點。

### 未知地點風險

使用者可能輸入店名、錯字、暱稱、路名。第一版若沒有地點資料，不可自動猜成附近景點。

### UX 風險

備援模式若做得太像第二套 planner，會破壞首頁既有主流程。第一版應低調、清楚、保守。

### 法律／責任風險

若網站顯示一定趕得上、末班幾點或這班車可搭，就有較高責任風險。未經官方資料確認，一律 Needs review。

## 9. 需要人工確認的問題

1. 第一版備援模式要不要出現在首頁？還是先只做資料規格？
2. 備援入口名稱要叫「備援模式」、「走歪救援」、「現在怎麼辦」還是其他？
3. 第一版是否允許使用者手動輸入目前地點？
4. 未知地點時，是否只要求使用者選最近車站／商圈，不做文字自由輸入？
5. 跨縣市點是否前台公開：屏東市、東港、恆春、墾丁？
6. 旗山、美濃是否算高雄旅遊圈，還是列遠郊備援？
7. 小港、楠梓、橋頭、岡山是否只做後台辨識點？
8. 是否要把「不建議但可自行決定」設為固定顯示，避免使用者覺得被阻止？
9. 是否接受第一版所有交通細節都標 Needs review？
10. 第二版是否要用 TDX 做離線資料驗證，而不是前端即時查詢？
